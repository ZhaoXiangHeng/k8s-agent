# 文档风格指南

## 概述

本 skill 定义了 k8s-agent 仓库的文档组织规范和写作风格，蒸馏自 `deploy-descision-agent` 仓库的文档实践，适配到本项目的技术栈和受众。

## 文档分层

写入文档前先判断内容属于哪一层：

| 层级 | 目录 | 受众 | 定位 |
|------|------|------|------|
| 入口 | `README.md`、`ARCHITECTURE.md` | 所有人 | 项目是什么、怎么跑起来 |
| 教程 | `docs/guides/` | 使用者 | 按步骤完成特定任务 |
| 产品 | `docs/product/` | 产品/面试官 | 业务需求、用户流程 |
| 架构 | `docs/architecture/` | 技术负责人 | 系统设计、数据模型 |
| 开发者 | `docs/developer/` | 接手代码的人 | 代码结构、技术原理、扩展方式 |
| 运维 | `docs/operations/` | 运维人员 | 部署、排错 |
| 参考 | `docs/reference/` | 所有人 | 查字段、查 API、查术语 |
| 安全 | `docs/security/` | 安全审查 | 认证、授权、威胁模型 |
| 设计 | `docs/superpowers/` | 历史沉淀 | 设计规格和实现计划 |

不要把开发者文档写进使用者教程，也不要把使用者操作手册写进架构文档。

## 每篇文档的结构要求

### 1. 开头段（必须）
第一段必须说明两件事：
- **这篇文档面向谁**
- **回答什么问题**

示例：
```
这篇文档面向第一次接手 k8s-agent 仓库的开发者，目标是在最短时间内回答三件事：
1. 这套系统是做什么的
2. 代码都放在哪里
3. 我应该怎么跑起来并开始改
```

### 2. 章节编号（必须）
使用 `## 1.` / `## 2.` 编号章节，方便导航和引用。

### 3. 区分状态（必须）
文档内容必须明确区分：
- **当前已实现**：代码中真实存在的功能
- **当前限制**：明确不支持的场景
- **推荐做法**：最佳实践
- **后续方向**：计划中但未实现的能力

不要把"设计愿景"写成"已经支持"。

### 4. 技术原理类文档使用"原则 + 落地方式"模式
```
## 1. 无状态 Agent
### 1.1 原则
Agent Server 不持久化任何会话历史。

### 1.2 落地方式
- `RunStream` 只消费 Backend 传入的 `context_messages`
- 不连接 PostgreSQL
- 每次请求结束后 agent loop 即销毁
```

## 写作风格

### 语言规则
- 文档正文使用中文
- 技术术语保留英文：JWT、gRPC、SSE、MCP、Eino、ReAct、ServiceAccount
- 代码、命令、API 路径、环境变量、配置键保留英文
- 日志中提到的字段名保留英文（如 `event=server_start`）

### 不要做的事
- 不要在文档中写大段代码实现细节（应指向源文件路径）
- 不要把多个受众的内容混在一篇文档里
- 不要写"显而易见"的背景知识
- 不要用 emoji 做装饰符号
- 不要在同一篇文档中重复其他文档已有的内容（应使用链接）

### 推荐做法
- 段落简短，一段只讲一个点
- 多使用表格比较选项/差异/边界
- 架构图使用 mermaid
- 引用源文件时给出具体路径
- 链接使用相对于 docs/ 的路径

## 子目录 README.md 要求

每个子目录下的 README.md（如 `docs/developer/README.md`）应包含：
1. 这组文档面向谁
2. 推荐阅读顺序
3. 每篇文档的一行简介

示例：
```
# 开发者文档中心
这组文档面向准备接手 k8s-agent 代码的开发者。

建议按下面顺序阅读：
1. 快速接手 — 理解目录、跑通命令
2. 架构总览 — 理解模块边界和执行链路
3. 技术原理 — 理解核心机制
```

## 修改文档时的检查清单

- [ ] 是否放在了正确的目录层级
- [ ] 开头是否说明受众和目的
- [ ] 是否区分了"已实现"和"未实现"
- [ ] 相关文档的入口链接是否更新
- [ ] 代码示例是否符合当前代码实际
- [ ] 是否与 AGENTS.md 中的项目协作规则一致
- [ ] 链接路径是否正确
