12：Backend 部署模板缺少 AUTH_MODE 和 KEYCLOAK_ISSUER 环境变量
   - Backend 代码已支持三种认证模式（dev/jwt/none），但 Helm 部署模板未注入对应环境变量
   - 导致生产环境实际运行在默认的 dev 模式，信任 X-Demo-User 头而不验证 JWT
   - 修复：在 values.yaml 增加 backend.authMode=jwt 和 backend.keycloakIssuer 配置，backend.yaml 模板注入 AUTH_MODE 和 KEYCLOAK_ISSUER 环境变量

13：Keycloak Issuer 与 Frontend Realm 不一致
   - 前端 config.ts 默认 realm=k8s-ai，但 backend 的 keycloakIssuer 之前指向 realms/master
   - JWT 验签时 iss 校验会失败，因为 token 中的 iss 与 backend 期望的不匹配
   - 修复：统一 values.yaml 中 Realm 为 k8s-ai，Keycloak Issuer 指向 `.../auth/realms/k8s-ai`

14：Keycloak 缺少路径前缀 (/auth) 配置导致 Traefik 路由失败
   - Traefik IngressRoute 用 PathPrefix(/auth/) 路由到 Keycloak，但 Keycloak 默认监听根路径 /
   - 请求 /auth/realms/... 被转发到 Keycloak 的 /auth/realms/...，但 Keycloak 只认得 /realms/...
   - 修复：Keycloak 启动参数增加 --http-relative-path=/auth，使 Keycloak 在 /auth 上下文路径下服务

15：Keycloak 未预配置 Realm 和 Client
   - Keycloak 以 start-dev 启动后只有默认 master realm，没有 k8s-ai realm 和 k8s-ai-frontend client
   - 需要手动通过管理控制台创建，无法自动化部署
   - 修复：创建 keycloak-realm-configmap.yaml，通过 --import-realm 在启动时自动导入 realm、client、角色和默认用户

16：前端 Dockerfile 缺少 Keycloak 生产环境编译变量
   - VITE_* 环境变量是 Vite 编译时变量，Dockerfile 构建时未设置
   - 构建出的镜像使用默认值 authMode=dev，keycloakUrl=http://localhost:8089，在 K8s 中完全不正确
   - 修复：Dockerfile 增加 ARG/ENV：VITE_AUTH_MODE=keycloak、VITE_KEYCLOAK_URL=/auth、VITE_KEYCLOAK_REALM=k8s-ai、VITE_KEYCLOAK_CLIENT_ID=k8s-ai-frontend、VITE_DATA_MODE=api

17：OIDC 回调 /auth/callback 被 Traefik 误路由到 Keycloak
   - 前端 OIDC 回调路径 /auth/callback 被 Traefik PathPrefix(/auth/) 规则拦截并转发到 Keycloak
   - Keycloak 无法处理该路径，导致 OIDC 登录流程在回调阶段失败
   - 修复：ingressroute.yaml 新增 Path(/auth/callback) 精确匹配，priority=300 > Keycloak 的 200，路由到 frontend

18：Keycloak 缺少 proxy-headers 配置导致 issuer 不确定
   - 无 --proxy-headers=xforwarded 时，Keycloak 根据请求 Host 头动态确定 issuer
   - 通过 Traefik 访问（外部 IP）和集群内访问（Service DNS）会生成不同的 issuer
   - 修复：增加 --proxy-headers=xforwarded 和 KC_HOSTNAME_URL 固定 issuer 为集群内部地址，KC_HOSTNAME_STRICT=false 允许外部访问

19：MCP Server 启动依赖 Backend 导致部署依赖环
   - 问题描述：远程部署时 backend-api、agent-server、mcp-server 形成启动依赖环，mcp-server 启动阶段长连接 Backend 失败后退出，服务无法全部就绪
   - 原因：mcp-server 在进程启动时连接 Backend Identity Service；Backend 又依赖 Agent Server，Agent Server 依赖 MCP Server，形成循环等待
   - 解决方案：将 mcp-server 到 Backend 的连接改为工具调用时短连接，NewClient 只保存地址，GetServiceAccount 每次请求时建立 gRPC 连接并在请求结束后关闭

20：前端管理控制台部署后仍显示 mock 数据
   - 问题描述：部署后测试用户管理、权限管理、模型分配页面仍显示本地 mock 数据，没有读取真实 Backend API
   - 原因：AdminConsole 子页面仍直接使用 `frontend/src/store/auth.ts` 的本地 mock store，`useAdminData` 也缺少管理员读取用户权限的接口调用
   - 解决方案：补充 `GET /api/admin/users/:id/permissions` 后端接口，前端管理控制台统一从 `useAdminData` 传入用户、权限、模型和绑定数据，并通过 API 操作函数保存变更

21：Keycloak 模式登录页仍显示 mock 默认账号
   - 问题描述：远程前端已编译为 `dataMode=api` 后，打开页面仍能看到默认账号 `admin/operator`，容易误判为仍在使用 mock 数据
   - 原因：`LoginPage` 静态引用 `frontend/src/store/auth.ts`，生产 bundle 中仍包含 mock 登录表单和默认账号提示
   - 解决方案：Keycloak 模式下登录页只显示统一登录按钮并触发 OIDC 登录，mock 登录逻辑改为开发模式动态加载，避免生产主 bundle 静态打入 mock 登录入口；同时为 nginx 入口页增加 `Cache-Control: no-store`，避免浏览器继续使用旧 `index.html`

22：公网 HTTP 访问 Keycloak 登录时报 `crypto.subtle.digest` 不存在
   - 问题描述：点击 Keycloak 登录按钮后，浏览器控制台报 `Cannot read properties of undefined (reading 'digest')`
   - 原因：公网 `http://120.55.84.39` 不是安全上下文，浏览器不开放 `crypto.subtle`，而前端 OIDC PKCE 登录默认使用 SHA-256 生成 `code_challenge`
   - 解决方案：前端检测不到 `crypto.subtle.digest` 时使用纯 JS SHA-256 生成 PKCE `code_challenge`，保证 HTTP 和 HTTPS 下授权请求与 Token 换取请求都携带完整 PKCE 参数

23：Keycloak 相对路径构造授权 URL 时报 `Invalid URL`
   - 问题描述：`VITE_KEYCLOAK_URL=/auth` 时点击登录报 `Failed to construct 'URL': Invalid URL`
   - 原因：浏览器 `new URL()` 直接接收 `/auth/...` 相对路径且未传入 base，会被判定为非法 URL
   - 解决方案：统一通过 `new URL(config.keycloakUrl, window.location.origin)` 解析 Keycloak base，授权、Token、Logout 地址都兼容 `/auth` 相对路径和完整 URL

24：远端 Keycloak `/auth/realms/k8s-ai` 返回 404
   - 问题描述：点击登录跳转到 `/auth/realms/k8s-ai/protocol/openid-connect/auth` 后 Keycloak 显示 Page not found
   - 原因：远端 Helm Chart 中 Keycloak Deployment 仍是旧模板，只执行 `start-dev`，没有启用 `/auth` 相对路径，也没有导入 `k8s-ai` realm
   - 解决方案：同步 Keycloak Helm 模板、realm 导入 ConfigMap 和 values；Keycloak 启动参数增加 `--http-relative-path=/auth`、`--import-realm`、`--proxy-headers=xforwarded`；同时移除 Client 强制 PKCE 配置，让公网 HTTP 降级登录可用，HTTPS 环境仍可由前端主动发送 PKCE；Backend 的 `KEYCLOAK_ISSUER` 改为与公网 Token `iss` 一致的 `http://120.55.84.39/auth/realms/k8s-ai`

25：Keycloak 回调后 `/api/me` 返回 401
   - 问题描述：Keycloak 登录完成回调后页面停在“正在完成登录”，浏览器请求 `/api/me` 返回 401
   - 原因：前端保存 Token 后立即调用 `loadMe`，但 React 闭包中的 `auth` 仍是旧的空 Token；另外 Backend 只读取顶层 `role` claim，未识别 Keycloak 标准的 `realm_access.roles`
   - 解决方案：前端将登录会话放入 React state，回调后用新 Token 直接请求 `/api/me`；Backend 从 `realm_access.roles` 中提取 `admin/operator`，并优先使用 admin 角色
